# 30. Response Compression Middleware in ASP.NET Core

Response Compression Middleware is a performance optimization feature in ASP.NET Core.

It reduces the size of HTTP responses before sending them to the client.

Smaller responses mean:

- Faster page loads
- Reduced bandwidth usage
- Better user experience
- Lower network costs

---

# What is Response Compression Middleware?

## Definition

Response Compression Middleware compresses the response generated by the server before sending it to the browser or API client.

Instead of sending a large response, it sends a compressed version.

The browser automatically decompresses it.

The user never notices the difference except that the response arrives faster.

# Enable Response Compression

## Step 1

Register the service

```csharp
builder.Services.AddResponseCompression();
```

---

## Step 2

Enable middleware


# Complete Example

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddResponseCompression();

var app = builder.Build();

app.UseResponseCompression();

app.MapControllers();

app.Run();
```

That's all.

# Configure Brotli

```csharp
builder.Services.AddResponseCompression(options =>
{
    options.EnableForHttps = true;
});
```

---

# Adding Providers

```csharp
builder.Services.AddResponseCompression(options =>
{
    options.Providers.Add<BrotliCompressionProvider>();

    options.Providers.Add<GzipCompressionProvider>();
});
```

---

# Configure Compression Levels

```csharp
builder.Services.Configure<GzipCompressionProviderOptions>(options =>
{
    options.Level = CompressionLevel.Fastest;
});
```

or

```csharp
builder.Services.Configure<BrotliCompressionProviderOptions>(options =>
{
    options.Level = CompressionLevel.Optimal;
});
```

---

# Compression Levels

| Level | Meaning |
|--------|----------|
| Fastest | Lower CPU usage, slightly larger compressed size |
| Optimal | Better compression, higher CPU usage |
| SmallestSize | Maximum compression, highest CPU usage |

# Best Practices

- Enable compression for text-based responses.
- Prefer Brotli when supported by the client for better compression ratios.
- Do not compress files that are already compressed (JPEG, PNG, ZIP, MP4, etc.).
- Measure CPU overhead in high-throughput systems before choosing the most aggressive compression level.
- Enable compression over HTTPS when appropriate.

# Frequently Asked Interview Questions

### What does Response Compression Middleware do?

It compresses HTTP responses before they are sent to the client, reducing payload size and improving application performance.

---

### Which compression algorithms are supported?

ASP.NET Core supports:

- Gzip
- Brotli

---

### Does the browser decompress the response?

Yes.

Modern browsers automatically decompress responses based on the `Content-Encoding` header.

---

### Should images be compressed using this middleware?

Generally, no.

Most image formats like JPEG and PNG are already compressed, so additional compression provides minimal benefit while consuming CPU.

# Interview Answer (2-Minute Version)

**What is Response Compression Middleware in ASP.NET Core?**

Response Compression Middleware compresses HTTP responses before sending them to the client, reducing the amount of data transferred over the network. It improves application performance, decreases bandwidth usage, and speeds up page loads, especially for text-based content such as JSON, HTML, CSS, JavaScript, and XML. ASP.NET Core supports **Gzip** and **Brotli** compression. Clients indicate supported compression algorithms through the `Accept-Encoding` request header, and the server responds with compressed content using the appropriate `Content-Encoding` header. The browser automatically decompresses the response before using it.
